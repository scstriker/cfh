# 术语合并系统 V3 PRD

## 1. 文档定位

- 文档类型：PRD + 实现映射
- 适用范围：仅针对“原子级制造术语”项目的当前版本改造
- 本轮产出：新增独立文档，不覆盖 `docs/implementation_plan_v2.md`
- 目标：将当前系统从“主稿驱动的多源嫁接”升级为“模板驱动的平等合并 + 质量守门”

---

## 2. 背景与现状

### 2.1 当前 V2 系统的核心机制

结合 `docs/implementation_plan_v2.md`、`inbox/开发文档/*_V2.docx` 与当前代码实现，V2 版本的核心机制仍然是：

1. 以某位专家作为主稿作者，其他专家定义围绕主稿增量嫁接。
2. 合并目标偏向“尽量保留多位专家贡献”，而不是“优先得到一条符合术语标准写法的定义”。
3. 审阅和导出围绕短语级来源溯源设计，但缺少独立的质量守门层。
4. 概念对齐虽已存在，但最终定义生成仍受主稿逻辑强约束。

这套机制已经能完成“多文档上传 -> AI 合并 -> 审阅 -> 导出”的完整流程，但当前输出在定义规范性、句式完整性、逻辑性和术语学一致性上已经出现明显问题。

### 2.2 当前版本的真实输出证据

以下问题均来自 `release/合并稿.docx` 的真实导出结果，不是抽象风险。

| 类型 | 词条 | 当前输出证据 | 已暴露的问题 |
|---|---|---|---|
| 句子拼接 | 原子级制造 | “通过规模化精准操控原子将制造精度或结构推进至原子水平使产品性能逼近极限的制造技术实现原子级别的可重复性……” | 多个来源短语直接串接，缺少逗号与句号控制，不是完整标准定义句。 |
| 重复堆叠 | 单原子 | “指在物理或化学系统中被孤立或独立操控的单个原子……以单个原子孤立形式存在……一个独立的原子单位……” | 同义信息反复叠加，没有去重和压缩。 |
| 术语失焦 | 原子线 | “由一个或多个原子按线性排列形成的一维结构……中性原子的外层电子从高能态跃迁至基态……发射的线状光谱” | 将“原子线”结构定义与“原子谱线”概念混在一起，发生概念漂移。 |
| 逻辑混乱 | 原子级抛光 | 定义中同时出现“加工技术”“装备”“表面粗糙度”“装备举例”等内容 | 将工艺、装备和效果混在同一条定义里，未按属概念与种差组织。 |
| 拼贴感过强 | 原子级构筑 | 多段“精准操控”“按预设结构”“高精度制造技术平台”“支持以原子或原子层等单元”顺次拼接 | 各专家原句被机械并列，读起来像语料堆叠，不像单一术语定义。 |
| 类型混合 | 原子级测量 | 同时出现“测量技术”“仪器设备”“时空能尺度”“检测任务装备集合” | 将“技术过程”与“装备集合”混为一体，术语边界不清。 |
| 缩写未规范 | 原子层颗粒包覆装备 | 定义中直接出现 “ALD” 及多套中英文别名串列 | 与“定义中不出现未展开缩写、不做别名堆叠”的规范目标冲突。 |

### 2.3 旧版系统规则废止清单

| 旧版假设 | 新版要求 | 处理方式 |
|---|---|---|
| 以主稿作者为骨架 | 以模板大纲为唯一结构骨架 | 废止 |
| 尽量保留每位专家表述 | 共识优先，偏差过滤，不追求“人人出镜” | 废止 |
| 合并成功的判断主要看来源覆盖 | 合并成功以定义质量和术语规范性为先 | 废止 |
| 审阅动作包含“采纳主稿原文” | 审阅动作改为“采纳指定专家原文” | 废止 |
| Phase 1 只上传专家文档 | Phase 1 必须同时上传模板大纲 | 废止 |
| 重点关注短语级溯源 | 保留溯源，但增加质量告警与质量校验层 | 保留并增强 |
| 纯前端、Gemini、分阶段审阅、docx 导出 | 继续保留 | 保留 |

## 3. 目标与非目标

### 3.1 目标

1. 输出的每条术语定义首先应是一条合格的国家标准术语定义，其次才是多源融合结果。
2. 用模板大纲统一目录结构、术语范围和概念锚点，取消主稿依赖。
3. 将“多专家融合”从“拼贴保留”改为“共识提炼 + 有价值补充 + 偏差过滤”。
4. 在现有纯前端架构内加入可解释的质量校验层，使问题在审阅前即可暴露。
5. 保留来源溯源能力，但不再以来源覆盖率牺牲定义质量。

### 3.2 非目标

1. 本轮不做通用标准平台化，不抽象成所有标准类型共用插件架构。
2. 本轮不扩展到试验方法标准、产品标准、管理标准等其他标准类型。
3. 本轮不要求将 `/inbox` 中 PDF 参考材料解析为逐条条文级引用库。
4. 本轮不要求在最终定义正文中直接嵌入 `[1][2]` 这类文献编号。

## 4. 输入基线与参考层级

### 4.1 输入材料优先级

| 层级 | 材料 | 角色 |
|---|---|---|
| L1 | `inbox/国家标准-原子级制造术语.docx` | 唯一结构骨架来源，决定章节结构、术语范围与目录顺序 |
| L2 | `inbox/基础术语/*.docx` 及其他 8 位专家草案 | 平等内容候选来源，提供定义素材与差异信息 |
| L3 | `inbox/百问百答.pdf` | 概念边界与术语解释参考语料 |
| L4 | `inbox/国标-增材制造术语.pdf`、`国标-纳米科技术语制造过程.pdf`、`国标-绿色制造术语.pdf`、`国标-智能制造服务通用要求.pdf` | 术语标准行文风格与概念边界参考语料 |
| L5 | `docs/implementation_plan_v2.md`、`inbox/开发文档/*_V2.docx` | 旧版产品假设、实施路径与保留约束参考 |

### 4.2 正向写法基线样本

以下样本用于约束目标定义风格：

1. 模板文档中的“原子级制造”：
   “通过规模化精准操控原子，将制造精度或结构推进至原子水平，使产品性能逼近极限的制造技术。”
   价值：单句完整、属概念明确、句内逗号自然。
2. 宋凤麒稿中的“单原子”：
   “指在物理或化学系统中被孤立或独立操控的单个原子，通常作为最小功能单元用于构建器件、催化位点或量子比特。”
   价值：句式完整，概念边界清晰，无重复堆叠。
3. 宋凤麒稿中的“原子级构筑”：
   “在原子或分子尺度上，通过直接操控和组合基本单元……按照预设的精确结构实现组装、排列或构建分子或材料的过程。”
   价值：前提条件、核心操作和作用对象顺序清晰。

## 5. 核心需求

以下 5 个核心问题来自 `inbox/notes.md`，每章均按“问题 -> 证据 -> 规则 -> 验收 -> 模块影响”组织。

### 5.1 P1 定义必须是完整句子，句内以逗号组织

#### 问题描述

当前合并结果经常将多个来源短语直接拼接，缺少清晰句法结构，无法作为标准定义直接使用。

#### 当前版本证据

- `release/合并稿.docx` 中“原子级制造”一条没有合理断句，多个从句直接顺接。
- “单原子”一条直到句末才出现句号，前文连续堆叠多个来源表达。

#### 新产品规则

1. 每条定义必须是一个完整的中文句子。
2. 句末只能有一个句号。
3. 句内分句使用逗号，不允许用分号拼接多个平行定义。
4. 定义应当能够直接替代术语名称使用。

#### 验收标准

- 所有合并定义均只有一个句号，且位于末尾。
- 不存在“多个来源片段直接串接但无标点组织”的结果。
- 抽检时，定义读起来应像标准条文，而不是来源摘录拼接。

#### 对实现的影响

- `lib/prompts.ts`：在合并提示中加入完整句规则。
- `lib/postprocess.ts`：新增 `sentence_form` 检查。
- `components/phase3/ReviewCard.tsx`：展示句式质量告警。

### 5.2 P2 多专家内容合并必须符合工艺与概念逻辑

#### 问题描述

当前系统常按“素材可保留则保留”的思路拼接内容，导致定义不遵循工艺先后、因果顺序或术语的属概念组织。

#### 当前版本证据

- “原子级抛光”同时混入加工技术、装备说明、表面粗糙度指标与装备举例。
- “原子级测量”同时混入测量技术、时空能尺度、仪器设备和检测装备集合。

#### 新产品规则

1. 合并定义必须遵循“属概念 -> 前提条件/环境 -> 核心操作 -> 作用对象 -> 结果/目标”的行文顺序。
2. 同一条定义不得同时混入工艺、装备、服务、评价指标等不同类别内容。
3. 若多个专家素材属于不同概念层级，应先做过滤，再合并。

#### 验收标准

- 抽检 5 条涉及工艺或设备的定义，行文顺序符合常规技术逻辑。
- 不再出现“设备定义中混入工艺过程”或“技术定义中混入设备集合”的结果。

#### 对实现的影响

- `lib/prompts.ts`：加入逻辑顺序与类型一致性约束。
- `lib/merger.ts`：在合并阶段优先按共识类型归类。
- `lib/postprocess.ts`：新增 `logic_order` 告警位。

### 5.3 P3 主框架不再以某位专家为主稿，统一以模板大纲为骨架

#### 问题描述

当前系统以 `primary_author` 作为骨架来源，导致术语范围、章节顺序和句式风格过度依赖某位专家稿。

#### 当前版本证据

- 现有代码和页面仍要求在 Phase 1 选择主稿作者。
- 审阅动作仍包含“采纳主稿原文”，说明主稿身份仍是主流程的一部分。

#### 新产品规则

1. `inbox/国家标准-原子级制造术语.docx` 是唯一结构骨架来源。
2. 8 位专家稿均为平等输入源，不再区分主稿与副稿。
3. 模板中有的词条必须进入处理范围；模板外词条必须作为孤儿词条等待挂载判断。
4. 审阅界面不得再使用“主稿 + 他稿”“仅主稿”等标签。

#### 验收标准

- Phase 1 必须上传模板文档。
- 不再要求选择主稿作者才能进入合并阶段。
- 导出目录结构与模板大纲一致。

#### 对实现的影响

- `store/reducer.ts`、`lib/types.ts`：`primary_author` 退出核心状态；新增 `template_outline` / `template_doc`。
- `app/phase1/page.tsx`：新增模板上传，移除主稿选择。
- `lib/exporter.ts`：按模板大纲组织导出结构。

### 5.4 P4 不是每位专家的话都必须出现在结果中

#### 问题描述

当前系统过度追求“保留所有专家贡献”，导致输出像拼贴画，无法稳定得到一条自然定义。

#### 当前版本证据

- “原子级构筑”“原子级修复装备”“原子级结构”等条目都表现出明显的多源堆叠。
- “单原子”条目同义重复严重，说明系统缺少去重与偏差过滤。

#### 新产品规则

1. 合并先识别共识，再决定补充信息，不以“每位专家都出现”为目标。
2. 明显偏离多数共识、重复价值低或表述质量差的内容应被排除。
3. 被排除的来源需要在结构化字段中记录原因，供审阅阶段查看。

#### 验收标准

- 输出定义读起来应是一条自然定义，而不是来源摘录列表。
- 被过滤内容可以在审阅阶段看到作者和理由。
- 抽检时，同义信息不得连续重复出现。

#### 对实现的影响

- `MERGE_RESPONSE_SCHEMA` 增加 `excluded_sources`。
- `MergeResult` 新增 `excluded_sources[{ author, reason }]`。
- `ReviewCard` 展示被过滤来源及理由。

### 5.5 P5 禁止循环定义和用术语解释术语

#### 问题描述

当前系统没有显式禁止循环定义，也缺少定义正文中对术语名中英文的检测。

#### 当前版本证据

- 现有提示词与输出结构中没有循环定义校验位。
- 当前导出稿中虽未普遍出现最典型的循环定义，但系统没有防线，属于已知缺口。

#### 新产品规则

1. 定义正文中不得直接出现被定义术语的中文名或英文名。
2. 若需要回指，使用“该技术”“该过程”“该装备”等泛指方式。
3. 循环定义属于硬质量问题，必须在审阅前暴露。

#### 验收标准

- 抽检定义正文，不出现术语中英文原词。
- 系统可自动检测并标记循环定义问题。

#### 对实现的影响

- `lib/postprocess.ts`：新增 `circular_definition` 检查。
- `MergeResult`：写入 `quality_flags`。
- 审阅界面：循环定义以高优先级告警展示。

## 6. 质量红线与校验规则

辅助问题 A1-A5 统一收口为质量红线和确定性校验规则。

| 编号 | 规则 | 默认要求 | 处理方式 |
|---|---|---|---|
| A1 | 定义长度 | 默认不超过 150 字 | 超出时打 `too_long` 告警 |
| A2 | 参考文献 | 在审阅/导出层管理，不要求正文直接插编号 | 导出时生成参考文献章节和引用项 |
| A3 | 从句长度 | 单个从句默认不超过 30 字 | 超出时打 `clause_too_long` 告警 |
| A4 | 缩写规范 | 不出现未展开缩写，不做英文别名堆叠 | 命中时打 `abbreviation` 告警 |
| A5 | 病句检测 | 病句只告警，不静默重写 | 打 `grammar` 告警，保留人工裁决 |

质量校验固定枚举：

- `sentence_form`
- `logic_order`
- `circular_definition`
- `too_long`
- `clause_too_long`
- `abbreviation`
- `grammar`

## 7. 工作流改版

### 7.1 Phase 0：概念对齐

- 输入：模板大纲 + 多位专家稿解析结果。
- 目标：以模板术语为锚点完成概念聚类，并识别模板外孤儿词条。
- 要求：
  - 不再围绕主稿作者建立概念簇。
  - `ConceptCluster` 需记录模板锚点信息，如 `template_term_id`、`in_template_scope`。
  - 保留 `suggested_chapter` 与 `mounting_reason`。

### 7.2 Phase 1：文档导入

- 上传内容改为“1 份模板大纲 + N 份专家稿”。
- 移除主稿选择器。
- 页面需要明确区分：
  - 模板文档
  - 专家输入文档

### 7.3 Phase 2：AI 逐条合并

- 合并原则改为“模板约束 + 共识优先 + 偏差过滤 + 规范校验”。
- 输出保留 `dimensions`、`merged_definition`、`segments`、`notes`。
- 新增：
  - `excluded_sources`
  - `quality_flags`
  - `reference_items`

### 7.4 Phase 3：审阅

- 审阅动作改为：
  - 采纳合并稿
  - 采纳指定专家原文
  - 手动编辑
  - 标记待议
- 审阅卡片新增：
  - 质量告警区
  - 被排除来源区
  - 参考文献信息区

### 7.5 Phase 4：导出

- 导出结构按模板大纲组织。
- 文末生成参考文献章节。
- 质量告警信息可展示或附录，但不阻断人工最终导出。
- 保留来源溯源能力。

## 8. 边界场景与期望行为

| 场景 | 期望行为 |
|---|---|
| 仅单一专家有定义 | 可直接采用该专家原文作为候选，但仍需经过格式与质量校验 |
| 模板有词条但所有专家都无定义 | 标记为“待补充”，进入人工处理列表 |
| 专家有定义但模板中没有 | 作为孤儿词条展示，给出挂载建议与理由，待人工确认 |
| 多专家冲突明显 | 共识不足时不强行拼接，保留争议说明并进入待议或人工裁决 |
| 命中循环定义 | 产生高优先级质量告警，不得静默通过 |
| 定义过长或从句过长 | 标记质量问题，但不直接自动重写 |
| 出现缩写或多套别名堆叠 | 标记质量问题，审阅时要求规范化 |
| 病句或逻辑顺序混乱 | 标记告警，交由人工审阅决策 |

## 10. 测试与验收

### 10.1 输出规范

- 定义必须是一个完整句子，且仅句末一个句号。
- 定义正文中不得出现被定义术语的中英文名称。
- 默认长度不超过 150 字。
- 不出现未展开缩写。

### 10.2 合并行为

- 多专家共识应输出一条自然定义，不得直接拼贴原句。
- 明显偏离共识的专家内容应被排除并记录原因。
- 单一来源定义可直接采用，但仍需经过格式校验。
- 模板内无定义词条需标记“待补充”。
- 模板外词条需给出挂载建议并等待人工确认。

### 10.3 审阅与导出

- 审阅界面支持采纳合并稿、采纳任一专家原文、手动编辑、待议。
- 导出文档保留来源溯源能力。
- 质量告警可见，但不阻断人工最终裁决。

## 12. 结论

V3 改造的核心不是继续提高“多源信息保留率”，而是重新定义什么叫“合并成功”。在本项目中，合并成功的首要标准应当是：以模板为骨架，从多位专家中提炼出一条合格、自然、可审阅、可导出的标准术语定义。来源溯源、专家贡献保留和交互式审阅都应服务于这个目标，而不是反过来主导定义生成。
